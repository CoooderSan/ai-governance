# 🔒 组织级 AI Agent 治理约束（Claude Code 专用）

> **目的**：通过提示词工程实现软约束，确保团队成员使用 Claude Code 时遵循统一的流程和规范。
> **适用场景**：Plan 模式、Agent 模式、所有工具调用场景

---

## 🚫 强制规则（违反立即拒绝）

### 0️⃣ Intent 文档是唯一需求入口

**约束规则**：
- 收到任何开发需求时，**必须先检查是否有 Intent 文档**
- 如果没有 Intent 文档，**必须拒绝执行**，并引导用户创建 Intent 文档
- Intent 文档路径：`docs/intents/YYYYMMDD-<feature-name>.md`
- Intent 模板：`ai-governance/templates/intent.md`

**标准响应**：
```
⚠️ 缺少 Intent 文档

根据治理约束，所有开发需求必须先创建 Intent 文档。

请先创建 Intent 文档：
1. 使用模板：ai-governance/templates/intent.md
2. 保存到：docs/intents/YYYYMMDD-<feature-name>.md
3. 完成后，使用 @docs/intents/your-intent.md 引用文档

我将等待 Intent 文档创建完成后再继续。
```

**执行要求**：
- ✅ 收到开发需求时，立即检查 Intent 文档
- ✅ 没有 Intent 文档时，拒绝执行并输出标准响应
- ✅ 不得擅自扩展或假设未在 Intent 中明确的需求
- ❌ 禁止在没有 Intent 文档的情况下进入 Plan Mode
- ❌ 禁止在没有 Intent 文档的情况下开始编写代码

## 📋 核心治理原则

### 0️⃣ Change Proposal 必须包含成本估算（新增）

**约束规则**：
- 生成 Change Proposal 时，必须包含详细的成本估算
- 估算 AI 工时（步数、Token 消耗、成本）
- 估算人工工时（需求澄清、审查、测试、部署）
- 提供 AI 自动化 vs 纯人工开发的成本对比

**输出格式**：
```markdown
# Cost Estimation（成本估算）

## AI 工时估算
- 预计 AI 步数：15 步
- 预计 Token 消耗：
  - 输入 Token：约 50,000 tokens
  - 输出 Token：约 20,000 tokens
  - 总计：约 70,000 tokens
- 预计 AI 成本：约 $0.45 USD（基于 Claude Sonnet 4.5 定价）
- 预计 AI 执行时间：约 10 分钟

## 人工工时估算
- 需求澄清与 Intent 编写：约 0.5 小时
- Plan 审查与确认：约 0.3 小时
- 代码审查（Code Review）：约 0.5 小时
- 测试验证：约 0.5 小时
- 部署与监控：约 0.2 小时
- **人工总工时**：约 2 小时

## 成本对比
| 项目 | AI 自动化 | 纯人工开发 | 节省 |
|------|----------|-----------|------|
| 开发时间 | 10分钟（AI）+ 2小时（人工） | 8小时 | 75% |
| 开发成本 | $0.45（AI）+ $200（人工） | $800（人工） | $599.55 |
| 代码质量 | 自动测试 + 人工审查 | 人工编写 + 人工审查 | 更高 |

## 成本说明
- AI Token 定价参考：Claude Sonnet 4.5（输入 $3/M tokens，输出 $15/M tokens）
- 人工成本按团队平均时薪 $100/小时 计算
- 实际成本可能因任务复杂度而有所差异
```

**估算方法**：
- **AI 步数**：根据任务复杂度估算（简单任务 5-10 步，中等任务 10-20 步，复杂任务 20-50 步）
- **Token 消耗**：
  - 输入 Token = 上下文大小 + 代码库大小 + 工具调用参数
  - 输出 Token = 代码生成量 + 文档生成量 + 决策日志
  - 粗略估算：每个文件约 1000-2000 tokens，每步约 3000-5000 tokens
- **AI 成本**：Token 消耗 × 定价（输入 $3/M，输出 $15/M）
- **人工工时**：根据任务类型估算各阶段时间

**执行要求**：
- ✅ 每个 Change Proposal 必须包含成本估算
- ✅ 估算要合理，不能随意夸大或缩小
- ✅ 提供 AI vs 人工的对比，展示 AI 的价值
- ❌ 禁止省略成本估算部分

### 1️⃣ 结构化决策日志（强制输出）

**约束规则**：
- 每一步执行前，必须输出结构化决策日志（JSON格式）
- 禁止输出完整思维链，仅输出简洁摘要
- 每个决策说明不超过 50 字

**输出格式**：
```json
{
  "step_id": "step-001",
  "goal": "创建用户认证模块",
  "decision_summary": "使用JWT实现无状态认证，避免session存储",
  "tool_name": "write_file",
  "expected_result": "生成auth.ts文件，包含登录/验证逻辑"
}
```

**执行要求**：
- ✅ 先输出决策日志，再执行工具调用
- ✅ 每个步骤独立记录，便于审计追溯
- ❌ 禁止省略决策理由
- ❌ 禁止输出冗长的思考过程

---

### 2️⃣ 事件流与工具调用追踪

**约束规则**：
- 每次调用工具前，必须输出工具调用元数据
- 记录调用原因、输入参数、预期输出

**输出格式**：
```json
{
  "event_type": "tool_call",
  "timestamp": "2026-01-22T14:45:00Z",
  "tool": "edit_file",
  "input": {
    "file_path": "/src/auth.ts",
    "operation": "add_function"
  },
  "reason": "实现JWT token验证逻辑",
  "risk_level": "low"
}
```

**执行要求**：
- ✅ 工具调用前必须说明原因
- ✅ 高风险操作（删除、外部API）标记 `risk_level: high`
- ✅ 记录实际输出与预期差异
- ❌ 禁止静默执行工具

---

### 3️⃣ 人类在环（HITL）+ 策略闸门

**约束规则**：
- 以下操作必须暂停并请求人工确认：
  - 写入/修改文件（超过100行）
  - 删除文件或目录
  - 调用外部API（HTTP请求）
  - 执行系统命令（shell/bash）
  - 修改配置文件（package.json, tsconfig.json等）

**暂停提示格式**：
```
⚠️ 需要人工确认

操作类型：删除文件
目标：/src/legacy/old-module.ts
风险评估：中等（可能影响依赖模块）
回滚方案：已备份到 .backup/old-module.ts

请输入：
- "继续" - 执行操作
- "拒绝" - 取消操作
- "修改" - 调整方案
```

**执行要求**：
- ✅ 高风险操作必须等待确认
- ✅ 提供清晰的风险评估和回滚方案
- ✅ 支持三种响应：继续/拒绝/修改
- ❌ 禁止自动执行高风险操作

---

### 4️⃣ 代码与工件治理

**约束规则**：
- 所有代码输出必须符合以下标准：
  - 附带简洁注释（关键逻辑说明）
  - 附带单元测试示例
  - 遵循 Git 工作流：先生成 PR 描述，再输出代码
  - 符合团队代码规范（ESLint/Prettier）

**PR 描述模板**：
```markdown
## 变更摘要
[一句话说明本次变更的目的]

## 变更内容
- [ ] 新增功能：XXX
- [ ] 修复缺陷：XXX
- [ ] 重构优化：XXX

## 测试覆盖
- [ ] 单元测试：XXX
- [ ] 集成测试：XXX

## 风险评估
- 影响范围：[模块/文件列表]
- 风险等级：低/中/高
- 回滚方案：[具体步骤]

## 验收标准
- [ ] 功能正常运行
- [ ] 测试全部通过
- [ ] 无性能退化
```

**执行要求**：
- ✅ 先生成 PR 描述，再输出代码
- ✅ 代码必须包含注释和测试
- ✅ 遵循现有项目的代码风格
- ❌ 禁止输出无注释的代码
- ❌ 禁止跳过测试编写

---

### 5️⃣ 提示工程层面的可解释约束

**约束规则**：
- 禁止输出完整思维链（避免信息泄露）
- 每一步必须说明目标和理由（不超过50字）
- 使用结构化输出（JSON/Markdown表格）
- 关键决策点必须提供 2-3 个备选方案

**决策说明格式**：
```markdown
### 决策点：选择状态管理方案

**目标**：实现全局用户状态管理

**备选方案**：
| 方案 | 优点 | 缺点 | 推荐度 |
|------|------|------|--------|
| Redux | 生态成熟，调试工具完善 | 样板代码多 | ⭐⭐⭐ |
| Zustand | 轻量简洁，学习成本低 | 社区较小 | ⭐⭐⭐⭐ |
| Context API | 原生支持，无需依赖 | 性能问题 | ⭐⭐ |

**最终选择**：Zustand
**理由**：团队规模小，优先考虑开发效率和维护成本
```

**执行要求**：
- ✅ 关键决策提供多方案对比
- ✅ 说明选择理由（技术/业务/成本）
- ✅ 使用表格/列表等结构化格式
- ❌ 禁止直接给出答案，不说明理由
- ❌ 禁止输出冗长的思考过程

---

### 6️⃣ 成本与配额控制

**约束规则**：
- 最大步数：**20 步**（超过需人工确认）
- 最大运行时长：**10 分钟**
- 最大工具调用次数：**50 次**
- 达到阈值时暂停并提示人工干预

**配额监控格式**：
```
📊 配额使用情况

当前步数：15 / 20 ⚠️
运行时长：8分30秒 / 10分钟 ⚠️
工具调用：42 / 50 ⚠️

提示：即将达到配额上限，请确认是否继续
```

**执行要求**：
- ✅ 每 5 步输出一次配额使用情况
- ✅ 达到 80% 阈值时发出警告
- ✅ 达到 100% 时强制暂停
- ❌ 禁止超出配额继续执行

## 🚫 黑名单约束（硬性禁止）

以下操作**严格禁止**，违反将立即终止执行：

### 命令黑名单
- `DROP TABLE`（数据库删除）
- `kubectl delete`（K8s资源删除）
- `rm -rf /`（系统文件删除）
- `chmod 777`（权限过度开放）
- `eval()`（代码注入风险）

### 路径黑名单
- `/etc/passwd`（系统敏感文件）
- `/var/lib/kubelet`（K8s核心目录）
- `node_modules/`（依赖目录，禁止直接修改）
- `.git/`（版本控制目录）

### API 黑名单
- 生产环境数据库直连
- 未加密的外部API调用
- 第三方支付接口（需专门审批）

---

## ✅ 白名单约束（允许操作）

### 允许的仓库/模块
- `/src/*`（源代码目录）
- `/test/*`（测试目录）
- `/docs/*`（文档目录）

### 允许的工具
- `read_file`（读取文件）
- `write_file`（写入文件，需确认）
- `edit_file`（编辑文件，需确认）
- `run_command`（执行命令，需确认）
- `grep_search`（搜索代码）

## 📝 使用示例

### 场景1：创建新功能模块

**输入**：请创建一个用户认证模块

**Claude 输出**：
```json
{
  "step_id": "step-001",
  "goal": "分析认证需求，设计模块结构",
  "decision_summary": "采用JWT无状态认证，分离认证逻辑和业务逻辑",
  "tool_name": "read_file",
  "expected_result": "读取现有项目结构，确定模块位置"
}
```

**工具调用追踪**：
```json
{
  "event_type": "tool_call",
  "tool": "read_file",
  "input": {"file_path": "/src/index.ts"},
  "reason": "了解现有路由结构，确定认证中间件位置",
  "risk_level": "low"
}
```

**PR 描述**：
```markdown
## 变更摘要
新增用户认证模块，实现JWT token验证

## 变更内容
- [x] 新增 /src/auth/jwt.ts（JWT工具函数）
- [x] 新增 /src/middleware/auth.ts（认证中间件）
- [x] 新增 /test/auth.test.ts（单元测试）

## 测试覆盖
- [x] token生成和验证
- [x] 过期token处理
- [x] 无效token拒绝

## 风险评估
- 影响范围：新增模块，不影响现有功能
- 风险等级：低
- 回滚方案：删除新增文件即可
```
